# Excel-to-Binary Game Data Pipeline

>> 엑셀 데이터를 **스키마 기반**, **FieldId 중심 바이너리**로 변환하는 
>> **CLI 기반 게임 데이터 파이프라인 도구**

---

## Overview

이 프로젝트는 게임에서 사용하는 대규모 정적 데이터를 효율적으로 관리하고 로드하기 위한 데이터 변환 파이프라인을 제공한다.
기획 및 밸런스 데이터는 엑셀에서 작성하고, 이를 자동으로 검증·변환하여 런타임에서 빠르게 로드 가능한 바이너리와 코드로 생성하는 것을 목표로 한다.

디자이너와 개발자의 작업 흐름을 분리하면서도 데이터 구조 변경에 안전하도록 설계되었으며, 빌드 및 CI 환경(Jenkins 등)에 쉽게 통합할 수 있는 CLI 기반 콘솔 도구로 제공된다.
이를 통해 반복적인 수작업, 런타임 파싱 비용, 구조 변경으로 인한 리스크를 최소화한다.

---

## Motivation

게임 개발 과정에서 엑셀 기반 데이터는 작성과 관리가 편리하지만,
런타임에서 직접 사용하기에는 다음과 같은 문제가 있다.

- 텍스트 기반 포맷으로 인한 느린 로딩 속도
- 구조 변경 시 코드 및 데이터 불일치 위험
- 런타임 파싱 비용과 GC 발생
- 클라이언트/서버 간 데이터 정의 불일치

본 프로젝트는 이러한 문제를 해결하기 위해
Excel → Schema → Code / Binary로 이어지는 명확한 변환 단계를 정의하고,
데이터 구조를 중심으로 한 자동화된 파이프라인을 제공한다.

---

## 왜 FieldId 기반인가

본 시스템은 필드 순서 기반이 아닌 FieldId 기반 바이너리 포맷을 사용한다.
이는 데이터 구조 변경에 대한 내구성과 장기적인 유지보수를 최우선으로 고려한 선택이다.

**순서 기반 바이너리의 문제점**
- 필드 추가/삭제 시 전체 구조가 변경됨
- 이전 바이너리와의 호환성 유지가 어려움
- 클라이언트와 서버 간 버전 불일치에 취약

**FieldId 기반 포맷의 장점**
- 필드 순서와 무관하게 데이터 로드 가능
- 알 수 없는 FieldId는 안전하게 `Skip`
- 누락된 필드는 `Default` 값으로 처리
- 신규 필드 추가 시 기존 바이너리와의 하위 호환성 유지

이 방식은 네트워크 프로토콜이나 직렬화 포맷(**Protobuf** 등)과 유사한 개념으로,
라이브 서비스 환경에서 데이터 구조가 지속적으로 변화하는 게임 개발에 적합하다.
